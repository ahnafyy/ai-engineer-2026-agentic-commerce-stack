What is agentic commerce?

A spectrum — not a binary. The question is how much of the loop the human still owns.

WHERE WE ARE NOW
Human in the loop
  • Agent surfaces products and pre-fills the cart
  • Human reviews and confirms before payment
  • Protocols (ACP, UCP, AP2) are built for this today
  • e.g. "find me cat cookies under $6" → agent presents, you tap Buy
Autonomous shopping
  • Agent decides what to buy, when, and how much
  • Executes checkout and payment without prompting
  • Runs on schedules or triggers — no human in the flow
  • e.g. auto-reorder household supplies when stock drops

The stack I'm walking through today is designed for the left column — but it's being built to support the right one.

[0:45] Before the protocols, let's be clear about where we actually are. Autonomous commerce — agent buys without asking you — is the end state everyone is building toward. But right now, humans are still in the loop. The agent surfaces the product, maybe pre-fills the cart, and you confirm. The entire ACP and UCP spec today is optimized for that handoff moment. AP2 exists precisely because you want a scoped mandate — the agent can't spend freely, it needs your pre-authorized token.

The first attempt: AI controls your browser

The obvious path to human-in-the-loop commerce — let the AI drive the same browser a human would use.

How it works
  • AI takes screenshots, reads the DOM, clicks elements
  • Navigates to merchant sites like a human would
  • Fills forms, adds to cart, proceeds to checkout
  • You watch or confirm at the end
Examples: Claude Computer Use chrome extension, Atlas browser
Why it's clunky
  • Slow — every step is a vision + reasoning round-trip
  • Brittle — a UI redesign breaks the whole flow
  • No structured data — scraping prices and stock is error-prone
  • CAPTCHAs, login walls, session timeouts
  • Not composable — every site is a bespoke automation

It works as a demo. It doesn't scale. The industry needed a better primitive.

[1:15] This was the first instinct — just let the AI do what a human does. Claude's computer use, Atlas, Operator all went this direction. The demos are impressive. The production reality is painful. You're one A/B test away from a broken checkout flow. The industry needed a protocol layer, not a smarter mouse.

ACP and UCP are both trying to own the same thing — how agents discover and buy from merchants. Both are open specs: ACP backed by OpenAI + Stripe, UCP backed by Google · ucp.dev.

[2:00] These 5 protocols compose into a complete stack. MCP handles tool access, A2A handles agent coordination, ACP/UCP handle commerce lifecycle, AP2 handles payment. I'll go deep on each.

MCP tool access

Agent discovers tools from a server, calls them with typed inputs, gets structured outputs. Stateless RPC.

// GET /tools — tool discovery
{
  "tools": [{
    "name": "product_search",
    "description": "Search the catalog",
    "inputSchema": {
      "type": "object",
      "properties": {
        "query":    { "type": "string" },
        "in_stock": { "type": "boolean" }
      },
      "required": ["query"]
    }
  }]
}
// POST /tools/call — execution
{
  "name": "product_search",
  "input": {
    "query": "cookies",
    "in_stock": true
  }
}

// Response
{
  "tool": "product_search",
  "count": 4,
  "results": [
    { "id": "prod_001",
      "name": "Paw Print Shortbread",
      "price": 4.99,
      "in_stock": true }
  ]
}
[4:00] MCP is the workhorse. It's how any LLM-powered agent accesses external capabilities in a standardized way. The tool schema is what gets passed to the LLM as its available functions. In our demo: 12 tools — product_search, inventory_check, apply_discount, checkout tools.

A2A agent coordination

Agent-to-agent JSON-RPC 2.0. Discovery via /.well-known/agent-card.json. Tasks have state: submitted → working → completed.

  • Agent card: capabilities, skills, extensions
  • Task lifecycle with state history
  • Extension slots → UCP hangs here
  • Streaming support via SSE
// POST /a2a
{
  "jsonrpc": "2.0",
  "method": "message/send",
  "params": {
    "contextId": "ctx_abc123",
    "message": {
      "parts": [{
        "kind": "text",
        "text": "add Paw Print Shortbread to cart"
      }]
    }
  }
}

// Agent card extension signals UCP support:
"extensions": [{
  "uri": "dev.ucp.shopping.checkout",
  "required": true
}]
[6:00] A2A is how the orchestrating agent talks to a merchant agent. The agent card is key — it tells the caller what the agent can do before any message is sent. UCP advertises itself as an A2A extension.

Facebook catalog / Instagram Shopping surface

This repo's catalog-sync service runs on a 60s schedule, reads from a single source JSON, and publishes all three. Engineering answer to protocol fragmentation.

[8:00] The intuition is wrong here. Most engineers assume ACP and UCP work like MCP — the agent calls product_search on your server and you return live results. That's not it. Both platforms run their own search index. You submit a feed, they ingest it, and their agent-side search queries *their* index — not yours. Your server is never in the hot path for discovery. This is the unglamorous engineering reality: three surfaces, three feed specs. The catalog-sync service in this repo does all three from one source of truth. AP2 agentic payment

A scoped, time-limited, single-use payment mandate. The agent carries this token — not a card number.

ChatGPT (ACP) — today payment only flows through shared payment token vendors (e.g. Stripe, Shopify Payments). AP2 is the spec for what comes next.
Gemini (UCP) — today payment only via Google Pay. Scoped mandate tokens are the roadmap. 
{
  "token_id": "tok_a3f9b2c1...",
  "sub": "user_001",
  "intent": "purchase:retail:paw_print_shortbread",
  "merchant_scope": "purrfect-bites.demo",
  "max_amount": 9.98,
  "currency": "USD",
  "expires_at": "2026-07-02T23:59:59Z",
  "single_use": true,
  "revocation_url": "https://pay.agent/revoke/tok_...",
  "user_consent_proof": "vc:credential:..."
}
  • sub — who authorized the agent
  • intent — exactly what it can buy
  • merchant_scope — only this merchant
  • max_amount — hard cap, no overruns
  • single_use — no replay attacks
  • revocation_url — kill switch
  • user_consent_proof — cryptographic human-authorized proof
[12:00] AP2 is still forming — this is the emerging pattern. The idea is the agent carries a cryptographically scoped token rather than raw payment credentials. The user consents once, the token constrains what the agent can spend, where, and when.

Why you should eval

Without evals, your agent will confidently do the wrong thing.

  • Chipotle's bot is helpful — just not for burritos
  • Your commerce agent will hallucinate prices, skip checkout steps, or call the wrong tools
  • You won't know until a customer does

Evals are how you know it works before it's live.

[15:00] Before I show the eval suite — why does this matter? Without evals, you have no signal. Your agent will hallucinate, skip steps, call the wrong tools. You'll find out at the worst moment. Chipotle's bot is technically helpful — it explained linked list reversal perfectly. But that's not what it was built for. Evals catch this.

How do you test this?

Behavior evals

Send canned prompts → assert correct MCP tools were called, checkout reaches COMPLETED state.

def test_bestsellers_tool_called():
    resp = send("what are your bestsellers?")
    assert "get_bestsellers" in tool_names(resp)

Protocol compliance

Validate MCP /tools schema, A2A JSON-RPC 2.0 structure, UCP checkout fields, ACP feed spec.

def test_ucp_native_commerce():
    feed = GET("/feed/ucp").json()
    for p in feed["products"]:
        assert "native_commerce" in p
